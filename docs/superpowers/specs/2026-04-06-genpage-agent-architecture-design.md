# Genpage Agent Architecture Design

**Date:** 2026-04-06
**Plugin:** model-apps
**Status:** Approved

## Problem

The current `/genpage` skill is a monolithic SKILL.md (~400 lines) that handles prerequisites, auth, requirements gathering, schema generation, code generation, deployment, and browser verification in one sequential flow. This makes it:

- Hard to maintain and evolve
- Impossible to parallelize (future multi-page generation)
- Difficult to extend with new capabilities (entity creation, app creation)

## Goals

1. Decompose `/genpage` into a slim orchestrator skill + specialist agents
2. Add Dataverse entity creation (tables, columns, relationships, choices, sample data)
3. Add model-driven app creation/selection
4. Architect for future parallel multi-page generation
5. Maintain backward compatibility — existing single-page flows should behave identically

## Non-Goals

- Cross-session memory bank (single orchestrated flow, not needed yet)
- Edit flows (existing edit workflow stays in the orchestrator for now)
- Multi-app generation (one app per invocation)

## Dependencies

- **Dataverse Skills plugin** (`microsoft/Dataverse-skills`) — soft dependency. Entity creation requires it. Without it, genpage works with existing entities only (current behavior).

## Architecture

### Current State

```
model-apps/
├── skills/genpage/SKILL.md    ← monolithic: prereqs + auth + requirements + schema + code + deploy + verify
├── references/
├── samples/
└── scripts/
```

### Proposed State

```
model-apps/
├── skills/
│   └── genpage/SKILL.md       ← slim orchestrator
├── agents/
│   ├── genpage-planner.md     ← requirements, discovery, plan doc, user approval
│   ├── genpage-datamodel-builder.md  ← DV entity creation via Dataverse plugin
│   └── genpage-page-builder.md       ← writes one .tsx file from plan + schema
├── references/                ← unchanged
├── samples/                   ← unchanged
└── scripts/                   ← unchanged
```

### Flow

```
User: /genpage "build me a candidate tracker"
          │
          ▼
   ┌─────────────┐
   │   /genpage   │  (orchestrator skill)
   └──────┬──────┘
          │
          ▼
   ┌─────────────────┐
   │ genpage-planner  │  Phase 1: Sequential
   │                  │  - Validate prereqs (node, pac)
   │                  │  - Auth + environment selection
   │                  │  - Gather requirements
   │                  │  - Detect what's missing (entities? app?)
   │                  │  - Present plan for approval
   │                  │  - Write genpage-plan.md
   └──────┬──────────┘
          │
          ▼ (only if entities need creating)
   ┌──────────────────────────┐
   │ genpage-datamodel-builder │  Phase 2: Sequential
   │                           │  - Create tables, columns, relationships, choices
   │                           │  - Uses Dataverse plugin tools (MCP + Python SDK)
   │                           │  - Ask about sample data → create if yes
   └──────┬───────────────────┘
          │
          ▼ (skill handles directly)
   ┌──────────────────┐
   │ App create/select │  Phase 3: Skill
   │ RuntimeTypes gen  │  Phase 4: Skill
   └──────┬───────────┘
          │
          ▼
   ┌─────────────────────┐
   │ genpage-page-builder │  Phase 5: Parallel-ready
   │  (×1 now, ×N later) │  - Reads genpage-plan.md + RuntimeTypes.ts
   │                      │  - Writes one .tsx file
   └──────┬──────────────┘
          │
          ▼
   ┌─────────────┐
   │   /genpage   │  Phase 6-8: Deploy + Verify + Summary
   │  skill       │
   └─────────────┘
```

### Working Directory Structure

The orchestrator creates a kebab-case folder from the user's description. All artifacts live flat in this directory:

```
candidate-tracker/
├── genpage-plan.md               ← plan doc (written by planner)
├── RuntimeTypes.ts               ← schema (generated by skill via pac CLI)
├── candidate-tracker.tsx         ← page (written by page-builder #1)
├── interview-dashboard.tsx       ← page (written by page-builder #2, future)
└── hiring-metrics.tsx            ← page (written by page-builder #3, future)
```

No nested hierarchy needed — each `.tsx` is a standalone single-file component with no cross-page imports.

## Agent Specifications

### 1. `genpage-planner`

**Purpose:** Gather requirements, discover environment state, produce an approved plan document.

**Invoked by:** `/genpage` skill via Task tool (sequential, Phase 1)

**Tools:**
- `Read`, `Write` — read references, write plan doc
- `EnterPlanMode`, `ExitPlanMode` — present plan for user approval
- `TaskCreate`, `TaskUpdate`, `TaskList` — progress tracking
- `AskUserQuestion` — gather requirements
- `Bash` — run `pac auth list`, `pac org who`, `pac model list`, `pac model list-languages`, `node --version`, `pac help`
- `mcp__dataverse__list_tables` — check which entities exist (optional, only if Dataverse plugin installed)

**Does NOT have:** Dataverse mutation tools, `Edit`, Playwright tools

**Steps:**

1. **Validate prerequisites**
   - `node --version` (must succeed)
   - `pac help` (version >= 2.3.1)
   - Commands run separately, not chained with `&&`

2. **Auth + environment**
   - `pac auth list` — check authentication
   - If no profiles: instruct user to `pac auth create`
   - If one profile: confirm it's active
   - If multiple: ask user to select via `AskUserQuestion`
   - `pac model list-languages` — detect configured languages (new pages only)
   - Report environment to user

3. **Gather requirements** (one question at a time via `AskUserQuestion`):
   - "Create new page(s) or edit an existing one?"
   - "Describe what you want built" (skip if description provided with command)
   - "Will the page use Dataverse entities or mock data?" (skip if already specified)
   - "Any specific requirements?" (styling, features, accessibility)

4. **Detect what exists vs what's missing:**

   a. **Entity detection** — check if Dataverse MCP `list_tables` tool is available:
      - **If available:** Call `list_tables`, compare user's requested entities against results. Mark each as "exists" or "needs creation".
      - **If not available and user needs new entities:** Inform user:
        > "Entity creation requires the Dataverse Skills plugin. Install it from microsoft/Dataverse-skills, or specify existing tables to continue."
        Proceed only with existing entities.

   b. **App detection** — `pac model list`:
      - 0 apps → ask user: "No model-driven apps found. Create a new one, or cancel?"
      - 1 app → confirm with user: "Found app [name]. Use this one?"
      - N apps → ask user to select one or create a new one

5. **Enter plan mode** and present:

   ```
   ## Genpage Plan

   ### Pages (N total)
   | Page | File | Purpose | Entities |
   |------|------|---------|----------|

   ### Data Strategy
   - Entities needed: [list]
   - Entities that exist: [list]
   - Entities to create: [list with columns, relationships, choices]
   - Sample data: [will ask after entity creation]

   ### App
   - Using: [app name + id] OR "Will create new app: [name]"

   ### Localization
   - [languages detected, if multiple]

   ### Design
   - [styling preferences, features, accessibility notes]
   ```

6. **Exit plan mode**, wait for user approval. Revise if changes requested.

7. **Write `genpage-plan.md`** to working directory containing:
   - User requirements (original prompt)
   - Working directory path
   - Plugin root path
   - Environment details (URL, app-id, languages)
   - Pages table (name, file, purpose, entities per page)
   - Entity specifications for entities that need creating:
     - Tables with column definitions (name, type)
     - Relationships (lookups, N:N) with cascade behavior
     - Choice/picklist columns with option values
   - Existing entity names (for RuntimeTypes generation)
   - Design preferences
   - Relevant sample references (which samples page builders should read)

### 2. `genpage-datamodel-builder`

**Purpose:** Create Dataverse entities specified in the plan using the Dataverse Skills plugin.

**Invoked by:** `/genpage` skill via Task tool (sequential, Phase 2 — only when plan says entities need creating)

**Tools:**
- `Read`, `Write` — read plan doc, write status
- `Bash` — run Python scripts (Dataverse SDK), PAC CLI commands
- `TaskCreate`, `TaskUpdate`, `TaskList` — progress tracking
- `AskUserQuestion` — ask about sample data
- Dataverse MCP tools: `list_tables`, `describe_table`, `create_table`, `update_table`, `create_record`

**Does NOT have:** `EnterPlanMode`, Playwright tools, `Edit`

**Steps:**

1. **Read `genpage-plan.md`** — extract entity specifications (tables, columns, types, relationships, choices)

2. **Check Dataverse plugin readiness** — verify `dv-connect` has been run (`.env` and `scripts/auth.py` exist). If not, instruct user to run `/dv-connect` first and stop.

3. **Create entities in dependency order:**
   - Independent tables first (no foreign key dependencies)
   - Then tables with lookups to already-created tables
   - For each table:
     - Create table via Dataverse MCP `create_table` or Python SDK
     - Add columns (string, int, decimal, datetime, bool, choice, etc.)
     - Wait for propagation delay (3-5s after table creation before adding columns)
     - Add relationships (lookups, N:N) via Python SDK
     - Add to solution via `pac solution add-solution-component`
   - Report actual logical names back (Dataverse may normalize them)

4. **Ask about sample data** via `AskUserQuestion`:
   > "Entities created successfully. Would you like me to add sample data?"
   >
   > Options: "Yes, add sample data" / "No, skip"

5. **If yes:** Generate and insert realistic sample records via Dataverse MCP `create_record` or Python SDK bulk create. Follow `dv-python-sdk` patterns for bulk import.

6. **Return result** to orchestrator:
   ```
   Entity creation complete.

   | Table | Columns | Relationships | Sample Records |
   |-------|---------|---------------|----------------|
   | cr_candidate | 8 | 1 lookup → cr_job | 10 |
   | cr_job | 5 | — | 5 |

   Ready for RuntimeTypes generation.
   ```

**Key constraints:**
- Follows `dv-metadata` patterns: environment-first (create via API, never hand-write solution XML)
- Uses Dataverse plugin tool hierarchy: MCP for simple ops, Python SDK for complex schema
- Never guesses column prefixes — reads actual publisher prefix from the environment
- Reports actual logical names back (they may be normalized by Dataverse)

### 3. `genpage-page-builder`

**Purpose:** Generate one complete `.tsx` page from the plan document and schema.

**Invoked by:** `/genpage` skill via Task tool (Phase 5 — one per page, all fired in parallel)

**Tools:**
- `Read` — read plan doc, RuntimeTypes.ts, references, samples
- `Write` — write one `.tsx` file
- `TaskCreate`, `TaskUpdate` — progress tracking

**Does NOT have:** `Bash`, `AskUserQuestion`, `Edit`, MCP tools, `EnterPlanMode`

Pure code-generation agent — all context comes from files.

**Steps:**

1. **Read `genpage-plan.md`** — extract this page's specification (purpose, entities, features, design preferences)

2. **Read `RuntimeTypes.ts`** (if Dataverse page) — extract verified column names, enum values, type definitions

3. **Read references:**
   - `${CLAUDE_PLUGIN_ROOT}/references/genpage-rules-reference.md` — full code generation rules
   - Relevant sample from `${CLAUDE_PLUGIN_ROOT}/samples/` (planner specifies which in the plan doc)

4. **Generate complete `.tsx`** following all rules from genpage-rules-reference.md:
   - React 17 + TypeScript + Fluent UI V9 exclusively
   - Single-file architecture (all components, utilities, styles in one file)
   - Only column names verified from RuntimeTypes.ts (never guess)
   - `makeStyles` with tokens for styling (no inline styles for static values)
   - WCAG AA accessibility (ARIA labels, keyboard navigation, semantic HTML)
   - Responsive design (flexbox, relative units, never `100vh`/`100vw`)
   - All async `dataApi` calls wrapped in try-catch
   - Lookup fields use `@OData.Community.Display.V1.FormattedValue` for display
   - No placeholders, TODOs, or ellipses
   - Components and utilities as separate top-level functions (no nested functions)
   - Localization support if multiple languages detected (per genpage-rules-reference.md)

5. **Write the `.tsx` file** to working directory

6. **Return result:**
   ```
   Page: Candidate Tracker
   File: <working-dir>/candidate-tracker.tsx
   Status: Written
   ```

**Key constraints:**
- **No MCP calls.** All context is in the plan doc and RuntimeTypes.ts
- **No questions.** Resolve all ambiguity from the plan doc
- **One file only.** Does not touch other pages' files
- **Uses exact values** from plan doc (entity names, column names, design preferences)

## Orchestrator Skill: `/genpage` Full Flow

The skill becomes a slim coordinator. Here is the complete phase sequence:

### Phase 0 — Create Working Directory

Derive a short folder name from the user's requirements, convert to kebab-case, create the folder, resolve absolute path.

### Phase 1 — Plan (sequential)

Invoke `genpage-planner` agent via Task tool. Pass:
- User's requirements (`$ARGUMENTS`)
- Working directory (absolute path from Phase 0)
- Plugin root path (`${CLAUDE_PLUGIN_ROOT}`)

Wait for planner to finish. Planner writes `genpage-plan.md` to working directory.

### Phase 2 — Create Entities (sequential, conditional)

Read `genpage-plan.md`. If entities need creating:
- Invoke `genpage-datamodel-builder` agent via Task tool
- Pass: path to `genpage-plan.md`, working directory
- Wait for completion

If no entities need creating: skip.

### Phase 3 — App Creation/Selection (skill does directly)

Read `genpage-plan.md` for app decision:
- If "create new": `pac model create --name "App Name"`
- If existing app-id: use it directly

Store app-id for Phase 6.

### Phase 4 — Generate RuntimeTypes (skill does directly, conditional)

If any page uses Dataverse entities:
```powershell
pac model genpage generate-types --data-sources "entity1,entity2,..." --output-file <working-dir>/RuntimeTypes.ts
```
Read the generated file to verify it succeeded.

If all pages use mock data: skip.

### Phase 5 — Build Pages (parallel)

Read `genpage-plan.md`, extract the pages table. For each page, invoke `genpage-page-builder` agent via Task tool. **Fire all invocations in a single message** for parallel execution. Pass each builder:
- Page name and target file name
- Path to `genpage-plan.md`
- Path to `RuntimeTypes.ts` (if Dataverse)
- Working directory
- Plugin root path (`${CLAUDE_PLUGIN_ROOT}`)

Wait for all builders to complete.

### Phase 6 — Deploy (sequential per page)

For each `.tsx` file produced:

**New pages:**
```powershell
pac model genpage upload `
  --app-id <app-id> `
  --code-file <file>.tsx `
  --name "Page Display Name" `
  --data-sources "entity1,entity2" `
  --prompt "User's original request summary" `
  --model "<current-model-id>" `
  --agent-message "Description of what was built" `
  --add-to-sitemap
```

**Mock data pages:** Same but omit `--data-sources`.

### Phase 7 — Verify (optional)

Ask user via `AskUserQuestion`:
> "Would you like to verify the page(s) in the browser using Playwright?"

If yes:
1. Navigate to each page URL: `https://<env>.crm.dynamics.com/main.aspx?appid=<app-id>&pagetype=genux&id=<page-id>`
2. Authenticate if needed
3. Structural verification (expected DOM elements based on page type)
4. Interactive testing (click a control, verify response)
5. Screenshot capture

If no: skip to Phase 8.

### Phase 8 — Summary

Present final summary:
```
## Genpage Complete

| Page | File | Entities | Status |
|------|------|----------|--------|
| Candidate Tracker | candidate-tracker.tsx | cr_candidate, cr_job | Deployed |

App: [app name] ([app-id])
Screenshots: [if verification was done]
Next steps: Share with team, iterate on design, create additional pages
```

## Edit Flow

The existing edit workflow (download existing page → modify → re-deploy) stays in the orchestrator skill for now. It does not go through the planner/builder agents because:
- It operates on a single existing file
- No entity creation is involved
- No plan document is needed
- The flow is simple: download → read → ask what to change → modify → deploy

If edit flows become more complex in the future (e.g., editing multiple pages, adding new entities to existing pages), they can be decomposed at that time.

## Dataverse Plugin Integration

The Dataverse Skills plugin (`microsoft/Dataverse-skills`) is a **soft dependency**:

| Scenario | Behavior |
|----------|----------|
| Dataverse plugin installed + entities exist | Planner calls `list_tables` to confirm. Datamodel builder skipped. Normal flow. |
| Dataverse plugin installed + entities missing | Planner detects missing entities. Datamodel builder creates them. Full flow. |
| Dataverse plugin NOT installed + entities exist | Planner skips entity check. User confirms entities exist. Normal flow (current behavior). |
| Dataverse plugin NOT installed + entities missing | Planner informs user: "Install Dataverse Skills plugin to create entities, or use existing tables." Falls back to existing-entities-only. |

## Migration from Current Skill

The refactoring preserves all user-facing behavior for the common case (existing entities, existing app, single page). Specifically:

**What stays the same:**
- All prerequisite checks (node, pac version)
- Auth flow (pac auth list/select)
- Language detection (pac model list-languages)
- Requirements gathering questions (same 4 questions)
- Plan presentation and approval
- Code generation rules (genpage-rules-reference.md, samples)
- RuntimeTypes generation and column name verification
- Deployment commands (pac model genpage upload with all flags)
- Browser verification via Playwright
- Final summary format

**What changes:**
- Skill delegates to agents instead of doing everything inline
- Plan is written to a file (`genpage-plan.md`) instead of held in conversation context
- Entity creation is a new capability (requires Dataverse plugin)
- App creation is a new capability (simple pac CLI)
- Architecture supports parallel page generation (future)

## Future Extensions

1. **Multi-page generation** — Already supported by architecture. Skill fires N `genpage-page-builder` agents in parallel. Each writes one `.tsx` file. Deployment loops through all files.
2. **Cross-session memory** — If workflows span multiple sessions (e.g., create entities today, build pages next week), add a `memory-bank.md` pattern similar to code-apps plugin.
3. **Edit decomposition** — If edit flows grow complex, introduce `genpage-edit-planner` and `genpage-edit-builder` agents (mirroring canvas-apps `canvas-edit-planner` + `canvas-screen-editor`).
